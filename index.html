  • Spring它是一个一站式的分层轻量级框架。
  • Spring的一些管理方面,应用场景
    1. core container

      3.spel它是spring提供的一个表达式语言

      1. 数据访问

      1.Spring本身提供spring mvc

    4. AOP
      AOP大部分情况下是使用动态代理来实现的。
    5. Test
      使用spring可以方便的进行测试
  • Spring的一些特点
    1. 方便解耦，简化开发
      Spring就是一个大工厂，可以将所有对象创建和依赖关系维护，交给Spring管理
    2. AOP编程的支持
      Spring提供面向切面编程，可以方便的实现对程序进行权限拦截、运行监控等功能
    3. 声明式事务的支持
      只需要通过配置就可以完成对事务的管理，而无需手动编程
    4. 方便程序的测试
      Spring对Junit4支持，可以通过注解方便的测试Spring程序
    5. 方便集成各种优秀框架
      Spring不排斥各种优秀的开源框架，其内部提供了对各种优秀框架（如：Struts、Hibernate、MyBatis、Quartz等）的直接支持
    6. 降低JavaEE API的使用难度
      Spring 对JavaEE开发中非常难用的一些API（JDBC、JavaMail、远程调用等），都提供了封装，使这些API应用难度大大降低

Spring 中的AOP

首先谈一下Aop,Aop(Aspect Oriented Programming)是指面向切口编程,通过预编译(静态)或者是动态代理技术(动态)来增强指定的方法(引用官方:实现程序功能的统一维护的一种技术),我们可以利用AOP技术将开发过程中的业务逻辑代码进行分离,使得业务逻辑代码便于维护,使我们代码的耦合度降低,提高程序的可复用性,维护性,也提升了开发的效率.

  • AspectJ是一个面向切面的框架，它扩展了Java语言。AspectJ定义了AOP语法所以它有一个专门的编译器用来生成遵守Java字节编码规范的Class文件。
  • 功能介绍
    *日志记录，性能统计，安全控制，事务处理，异常处理等等
  • 主要意图
    • 将日志记录，性能统计，安全控制，事务处理，异常处理等代码从业务逻辑代码中划分出来，通过对这些行为的分离，我们希望可以将它们独立到非指导业务逻辑的方法中，进而改变这些行为的时候不影响业务逻辑的代码。
  • 与OOP(面对对象编程)的区别
    *总之我们的OOP是指对某一个代码的片段,细致到类中的莫一个方法,而OOP是对类的规范
    • 引用官方话
      • OOP（面向对象编程）针对业务处理过程的实体及其属性和行为进行抽象封装，以获得更加清晰高效的逻辑单元划分。
      • 而AOP则是针对业务处理过程中的切面进行提取，它所面对的是处理过程中的某个步骤或阶段，以获得逻辑过程中各部分之间低耦合性的隔离效果。这两种设计思想在目标上有着本质的差异。
        *换而言之，OOD/OOP面向名词领域，AOP面向动词领域。
  1. AOP的相关术语
    1. 目标对象Target
      指的是需要被加强的类,Spring中的AOP是使用代理方式实现,所以永远是被代理的一方

复习笔记

##Strust2

  1. Java 中常用的框架

    1. SSH
      • Strust2
      • Spring
      • Hibernate
    2. SSM
      • Spring MVC
      • Spring(Spring boot)
      • Mybatis(Ibatis)

  2. 框架运行的流程(Strust2)

    Struts2是一个基于MVC设计模式的Web应用框架，它本质上相当于一个servlet，在MVC设计模式中，Struts2作为控制器(Controller)来建立模型与视图的数据交互
    1. 客户端发送的请求
    2. 请求被Filter拦截器阻拦(StrutsPrepareAndExecuteFilter)
    3. 接着拦截器询问ActionMapper是否需要调用Action
    4. 如果需要调用,将把请求的处理交给ActionPooxyAction(代理)
    5. ActionPooxyAction(代理)加载框架的配置文件,找到需要调用的Action
    6. ActionPooxyAction(代理)创建一个ActionInvocation的实例
    7. 在ActionInvocation的前后过程,会设计到拦截器的加载
    8. 一旦Action执行完毕,ActionInvocation就会找strust.xml文件找寻结果

  3. 过滤器和拦截器的区别

    1. 过滤器可以拦截所有请求,但是拦截器只能拦截Action
    2. 拦截器是基于Java的反射机制,而过滤器是基于回掉函数
    3. 拦截器可以访问Action的上下文,值栈,过滤器不行
    4. 过滤器是基于Servlet容器的,拦截器不需要
    5. 生命周期:过滤器的生命周期是只能在容器初始化的时候才会调用,但是拦截器器可以多次启用
    6. 执行顺序 过滤器→拦截器→Action请求处理→拦截器→过滤器

  4. Strust2.xml配置详解

    1. package
    • name__唯一标识,不可重复__
    • namespac 和Acton中的Name标签组合起来可以构成项目的访问路径
    • exends 继承,可以继承于别的包,通常使用Strust.default
    • absract 抽象,表明是用于继承的类
    1. Action
    • name 和Package中的Name标签组合起来可以构成项目的访问路径
    • class 内容是被Action对象的全类名
    • method 用于指定是类中的某一个方法
    1. result
    • type 指定是使用哪一种方式经行跳转(如forward,重定向,默认使用重定向),不需要加工程名
      • dispatcher 转发到Jsp
      • chain Actio转发
      • redirectAction 重定向到Action
      • rediret
    • name 用于和Action中方法的返回值进行匹配

  5. Action的创建方式

    1. 创建一个普通的pojo类 什么也不做,不需要继承也不需要实现(优点是耦合度很低,但是什么都需要我们自己完成)
    2. 实现接口 Action 耦合度底,在Action接口中定义了五种strtus2的逻辑视图名称，它们可以被框架识别。缺点:功能还需要我们来完成
      • 接口自带五个常量(字段)
        • Error 错误试图
        • input 是Strust2中拦截器代理发生错误的试图
        • login 一个登陆视图,可以在权限操作中使用
        • none 它代表的是null,什么都不会做
        • success 成功视图
    3. 继承ActtionSupp类 优点:具有了丰富的功能，表单校验，错误信息设置，国际化功能。缺点:耦合度高我们在开发中使用继承ActionSupport这种方案比较多。

  6. 使用Strust封装数据

    1. 属性驱动
      • 在类中创建字段,但是需要和页面中的name名字一致,并且提供getter方法
      • 如果是一个get的一个对象,我们需要在页面中将名字设置为 对象名+对象的属性名称 (不建议使用)
    2. 模型驱动
      • 我们需要实现一个接口 ModelDriver 然后在类中实现方法getModel方法
      • 需要在类中new出该类

  7. Stust2如何获得ServletApi

    1. 使用注入,例如实现接口HttpServletRequstAware 然后在类中声明出来需要的对象,然后在重写的方法中实现.this.request = request 即可,原理是拦截器中的一个拦截器赋予的
    2. Stusst2api获取
      • ServletActoinContext获取 request , session , response , ServletContexxt

  8. OGNL表达式

    • OGNL是Object-Graph Navigation Language(对象图导航语言)的缩写，它是一种功能强大的表达式语言，通过它简单一致的表达式语法，可以存取对象的任意属性，调用对象的方法，遍历整个对象的结构图，实现字段类型转化等功能。它使用相同的表达式去存取对象的属性。
    1. Strust 集成了Ognl表达式,Ognl本来也是一个项目,可以单独使用
    2. Ognl使用必须有三要素
      • Ognl表达式
      • Ognl上下文,OgnlContext
      • Root
    3. 取值 如果在非Root区中取值是需要加#号的
  9. ##值栈(ValueStatck)
    • 是一个容器,然后是用于储存数据,和Ognl配合用于数据显示,里面包括了Action对象的所有数据操作和Tomcat的数据(requst,Session等等)
    • 一个request → Action → ActionContext → ValueStatck
    • 存储机构
      1. root区
        1. CompoundRoot(其实是一个ArrayList的子类),中间装了Action的数据
          • 自动填充 既整个Action对象,包括了Model也会被填充进去
          • 手动加入的 可以使用set方法或者是put,我们一般使用set
          • 他的进出方式是属于一个压栈,
        2. context 一个Map集合
          • key是request,session,application(servletcontext),value是一个map集合,里面装了名字和具体的属性
      2. 值栈的获取
        1. 使用ActionContext.getContext.getValueStatck
        2. 从request中获取值栈(因为就是存储到request中)
      3. 值栈中值的获取
        1. 在root区中,如果是使用set方法存入的话,直接使用key的名字即可
        2. 如果是root区中按照索引取,例如 <S:property:’[0].top’>
        3. 在Context区可以使用 <S:property:’#key’>
        4. 在context区中获取域对象的值 <S:property:’#request.key’>
        5. 在context区中拿到root区的值,这两个区是相互引用的,可以在Context区中调用到root区的值
      4. 拦截器介绍(Interceptor)
        • Struts2中的interceptor它是基于spring aop思想，而aop思想它本质上是通过动态代理来实现。我们strtus2的拦截器它主要是拦截Action的操作，在action的执行前或执行后进行一些其它的功能操作。拦截器链（拦截器栈）简单说，就是可以将多个拦截器形成一个链，在访问它们时依次访问。
        • 如何去生成一个拦截器?
          • 让一个类实现Interceptor接口(strust中的Xwork包,不是原生的拦截器的接口),然后重写里面的方法.
            • strust2中配置格式如下在标签中


            • 值得注意的是需要在Action中引用拦截器再次引用strust-default,因为引用外部的拦截器会使原本的拦截器失效
            • 或者我们可以在拦截器中声明了引用了默认拦截器,这样我们就不需要在Action中再次引用,但是我认为这样不太灵活
          • 继承一个抽象类(MethodFilterInterceptor)并且重写其中的方法doIntercept实际上也是实现Interceptor接口
            • 实现该接口会有一个类ActionInvocation,调用其中的invoke方法可以将程序放行
